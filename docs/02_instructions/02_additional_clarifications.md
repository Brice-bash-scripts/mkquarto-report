# Second Additional Clarifications

1. Project Root Naming Behavior
Answer: The script should create the target folder using the slugified title (Option B).

Reasoning: The slugified title ensures that folder names are consistent, URL-friendly, and avoid issues with special characters or spaces. For instance, "Real Estate Prediction" would be turned into "real-estate-prediction". This approach makes it easier to reference the project folder and ensures compatibility across different systems.

Note: The project folder name should not always be "technical_report," as allowing a custom name provides more flexibility for users.

2. Canonical Include Order in report.qmd

Answer: The final include sequence should be:

Cover page (e.g., content/00_frontmatter/cover.qmd)
Table of contents (using Quarto’s built-in TOC handling, not hard-coded in the report.qmd)
Sections 01 to 09 (content files for each section, such as content/section_01/01_0_executive_summary.qmd)
Figures and Appendices (e.g., content/figures/figures.qmd, content/appendices/appendices.qmd)
References (e.g., content/section_10/10_0_references.qmd)
Reasoning: The cover page should come first, followed by the table of contents, and then the main sections of the report. Figures and appendices should be included after the main sections, as they are usually referenced within the report body. References should be placed at the end of the document, after the main content.
Note: The TOC should be auto-generated by Quarto based on the configuration, not hard-coded inside the report.qmd, to ensure it reflects the actual structure of the report and can be easily updated as sections are added or removed.

3. Table of Contents Strategy
Answer: The TOC should be generated only via the _quarto.yml settings.

Reasoning: Generating the TOC via the YAML configuration ensures that it is consistent and automatically reflects the sections and structure defined in the report. Hard-coding a TOC in the report.qmd could lead to duplicates or missing entries if the sections change, and it could interfere with the automatic rendering that Quarto provides.

Note: Ensure the _quarto.yml file is correctly set up to generate the TOC for PDF output, and don't mix it with manual LaTeX commands to avoid conflicts.

4. Force Overwrite Safety Model
Answer: Option A: maintain a manifest file of generated paths.

Reasoning: The manifest file will track which files were generated by the script, making it easier to determine if a file needs to be overwritten. This ensures that only the files generated by the script (like templates or default configurations) are overwritten, and user-generated content is left intact.

Manifest File: The script can create and maintain a manifest file (e.g., generated_files_manifest.txt) that lists the paths of all files it generates. This allows for consistent checking and behavior when the --force flag is used. When the --force flag is used, the script can refer to this manifest to determine which files to overwrite, ensuring that user-generated content is not accidentally overwritten.

5. Template Extension Migration in Docs
Answer: The template instruction files should remain as .md files for documentation purposes while the script outputs .qmd files.

Reasoning: Keeping the template instructions as .md files allows them to serve as guides for users while ensuring the generated report uses .qmd files (which are Quarto-compatible). This avoids confusing users by mixing file types in the template docs, and the .md format for instruction files will be more familiar to users.

Note: The script should only generate .qmd files in the project structure, but the instructions and documentation for the template can remain in .md format.

6. Render Validation Scope
Answer: The script should support both full render and fast validation mode.

Reasoning: A full render is necessary to ensure the entire report, including PDF generation, is functioning correctly. However, for CI and development, a fast validation mode (which checks just the structure and content, without generating a PDF) will save time. The fast mode should be an option for users to run during iterative development.

For full render failures: If the full render fails due to missing PDF dependencies (like LaTeX), the script should still scaffold the project and exit with a non-zero status. This ensures the user has a valid project structure to work with and can install the necessary dependencies or troubleshoot the rendering issue.
